Replace the ZStack offset approach with true conditional rendering using `.transition(.asymmetric())` as documented in ISS-040 research.

In CategoryGrowthListView.swift body (lines 22-53):
- Keep ZStack structure but simplify conditional rendering
- Remove background dimming overlay (lines 23-33) - not needed for push animation
- Change line 36-42: Keep `.transition(.asymmetric(insertion: .move(edge: .leading), removal: .move(edge: .leading)))`
- Change line 43-50: Keep `.transition(.asymmetric(insertion: .move(edge: .trailing), removal: .move(edge: .leading)))`
- Line 52: Keep `.animation(.easeInOut(duration: 0.3), value: selectedCategory)`

**Critical change:** Ensure ONLY ONE view exists at any time:
```swift
if computedSelectedCategory == nil {
    categoryListView
        .transition(.asymmetric(
            insertion: .move(edge: .leading),
            removal: .move(edge: .leading)
        ))
} else {
    categoryDetailView(for: computedSelectedCategory!)
        .transition(.asymmetric(
            insertion: .move(edge: .trailing),
            removal: .move(edge: .leading)
        ))
}
```

**Why this works (from ISS-040 research):**
- Only ONE view renders at a time - no overlap possible
- List exits left while detail enters right - synchronized push effect
- SwiftUI handles transition timing automatically - no manual offsets needed
- Back button triggers reverse: detail exits left, list re-enters from left
- No transparency fighting because only one view exists

**What to avoid:**
- Do NOT use ZStack with both views existing simultaneously with opacity/offset tricks
- Do NOT use `.offset()` modifiers - SwiftUI `.move(edge:)` transitions handle positioning
- Do NOT add background dimming - it interferes with push animation perception
