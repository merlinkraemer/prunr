<task type="auto">
  <name>Fix UAT-003: Refresh Snapshots command does nothing</name>
  <files>Prunr/Views/MainView.swift</files>
  <action>
The refreshAction is wired via .focusedValue(\.refreshAction) in MainView.swift:86-90 but the closure wraps a Task that doesn't wait for completion. The action also doesn't provide any user feedback.

Current implementation:
```swift
.focusedValue(\.refreshAction) {
    Task {
        await viewModel.loadSnapshots()
    }
}
```

Fix:
1. Keep the async Task wrapper (required for await)
2. After loadSnapshots() completes, re-run autoSelectSnapshots() to restore selection
3. If selected snapshots exist, re-trigger compareSnapshots() to refresh deltas
4. Optional: Consider adding brief visual feedback (e.g., flash or subtle animation)

The key issue is that loading snapshots clears and replaces the array, which may deselect current selections. Need to preserve or restore selection state.

<task type="auto">
  <name>Fix UAT-001: Window can be resized to very small size</name>
  <files>Prunr/PrunrApp.swift</files>
  <action>
The PrunrApp.swift:26 uses `.windowResizability(.contentMinSize)` which relies on the content's intrinsic minimum size. For SwiftUI content with flexible sizing (Lists, VStacks), this doesn't enforce a meaningful floor.

Fix: Replace `.windowResizability(.contentMinSize)` with `.frame(minWidth: 600, minHeight: 400)` on the WindowGroup content to enforce explicit minimum dimensions.

Modified implementation:
```swift
WindowGroup {
    MainView()
        .frame(minWidth: 600, minHeight: 400)
}
.defaultSize(width: 800, height: 600)
```

This ensures the window cannot be resized smaller than 600x400, which is reasonable for the two-column picker layout + delta list.
  </action>
  <verify>
1. Build and run app
2. Try dragging window corner to shrink
3. Verify window stops at approximately 600x400
4. Verify window can still be resized larger
  </verify>
  <done>Window has minimum size constraint of 600x400, cannot be resized smaller</done>
</task>
