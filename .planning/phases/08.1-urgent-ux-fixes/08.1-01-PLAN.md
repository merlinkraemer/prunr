  <name>Task 2: Improve GB meter real-time updates (ISS-032)</name>
  <files>Prunr/Services/MenuBarManager.swift</files>
  <action>
Reduce the cache interval for disk space updates to improve real-time responsiveness of the GB meter in the menu bar.

Current implementation (around line 66-67): Cache prevents updates more frequent than 5 seconds (`lastFreeSpaceUpdate`).

Options to implement:
1. **Reduce cache interval** (simpler): Change from 5s to 1-2s for more frequent updates
2. **Trigger on FSEvents** (more reactive): Call updateFreeSpace() when FSEvents detects changes

Recommended approach: Start with #1 (reduce to 2 seconds), as it's simpler and less likely to cause performance issues. If still not responsive enough, add FSEvents trigger in a follow-up.

Implementation:
- Find `updateFreeSpaceIfNeeded()` method
- Locate cache interval constant/check (likely comparing Date difference to 5 seconds)
- Change threshold from 5 seconds to 2 seconds
- Add comment explaining the reduced interval for real-time UX

Why 2 seconds: Balance between real-time feedback and not spamming disk space API calls. macOS disk space checks are relatively fast, but we don't want to call them every frame.
  </action>

3. Verify it's been reduced to 2 seconds

The monitoring path header should be replaced (not coexist) when drilling down into category details. Currently both headers show separately creating visual confusion.

Current state:
- MenuBarView.swift has `mainHeaderView` (lines ~141+) with drive bar + monitoring path
- CategoryGrowthListView.swift has `CategoryDetailHeader` (line ~107) with back button + category name
- Both appear simultaneously during drill-down

Desired state:
- Main view: Show mainHeaderView (drive bar + monitoring path)
- Detail view: Show ONLY CategoryDetailHeader (back button + category name)
- The mainHeaderView should be hidden/replaced when drilling down

Implementation approach:
1. In MenuBarView.swift, add a condition to hide mainHeaderView when drill-down is active
2. Need to detect drill-down state - CategoryGrowthListView has @State private var selectedCategory
3. Options:
   a) Pass selectedCategory binding up to MenuBarView (requires architecture change)
   b) Use @Binding in CategoryGrowthListView to communicate state up
   c) Add isDrilledDown state to MenuBarManager (cleanest for Observable pattern)

Recommended: Option C - Add `var isDrilledDown = false` to MenuBarManager, set it in CategoryGrowthListView when selectedCategory changes.

Steps:
1. Add `var isDrilledDown = false` to MenuBarManager
2. In CategoryGrowthListView, add `@Bindable var manager: MenuBarManager` if not present, or access via environment

4. In MenuBarView mainCategoryView, conditionally show mainHeaderView only when !manager.isDrilledDown
5. Ensure CategoryDetailHeader is positioned at the same location as mainHeaderView for seamless replacement

Why this approach: Uses existing @Observable pattern, minimal changes, state management stays clean.
