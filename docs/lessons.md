# Lessons

- Audit fixes must include the integration path, not just the lower-level service behavior. If a service can return an escalation signal like `needsFullScan`, verify the caller actually handles it end-to-end.
- Permission gating for macOS scanning must be validated against the protected locations the product actually touches, not a single surrogate probe like the TCC database. If scan-time prompts still appear, the onboarding permission state is wrong.
- On macOS, removing explicit permission probes is not enough. Startup watchers and any background filesystem monitors must also stay disabled until onboarding is complete and a real baseline exists, or the app will still trigger protected-folder prompts on launch.
- For UI affordance fixes, verify the exact user-triggered runtime path. Removing one trigger is not enough if the shared downstream path can still escalate into the old behavior, and shape-level gestures are less reliable than explicit button interactions in dense SwiftUI layouts.
- A stable soak is not enough for scan-pipeline validation. Always include a functional freshness check that creates or modifies a file after the baseline and proves the working set, category totals, and UI update.
- Dirty FSEvents batches that arrive while a full scan is already running must not schedule another full scan. The active full scan supersedes dirty-root fallback; otherwise each scan can arm the next scan after the dirty backoff delay.
- Full-scan cleanup must account for process death, not only Swift cancellation. A killed app cannot run ScanService's catch handler, and later summary backfills can make a partial snapshot look complete. The durable invariant is that completed scans rebuild the working set at the snapshot timestamp, so startup cleanup should remove snapshots newer than the working set.
- A scheduled debounce task must not remain the ownership handle for work after it starts executing. Clear or transfer ownership before doing the async work, or later events can cancel the active refresh and leave pending state churning forever.
- Fast UI placeholders must not satisfy the same "loaded" invariant as enriched DB state. If quick inventory omits growth stories or snapshot identity, panel-open enrichment still has to run.
- Full-rescan cooldowns should use the user-configured scan interval, not just an internal storm debounce. If a dirty event is intentionally dropped during cooldown, also drop the visible pending reason.
- After a valid baseline exists, watcher arming must not be gated by full protected-folder traversal permission. Incremental event handling can monitor the reachable root while full scans and reconciliations stay separately permission-gated.
- Single-file FSEvents updates and removals must not use subtree replacement SQL on a multi-million-row working set. Use exact-row updates first, and make subtree deletion query `paths` by indexed path range before joining `workingSetEntry`.
- Monitor probes must measure the same bytes the app records. If Prunr stores allocated disk usage, the probe should use allocated blocks and avoid compressible payloads instead of assuming logical file size equals growth.
- A fast panel-open cache refresh must be identity-preserving. If quick data has the same category totals as the enriched rows already on screen, do not assign a new `allCategories` array or strip growth metadata just because the hosting view was recreated.
- Do not kill tmux windows by broad session/window index during a live collaboration. Create uniquely named detached monitor windows and only target those exact names/PIDs for cleanup.
